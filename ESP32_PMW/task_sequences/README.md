# task_sequences/

SPIFFS-uploaded payload for `[env:experiment]` (`src/main_experiment.cpp`).
**Not** the top-level `data/` directory — that one holds captured
experiment results (CSVs, plots, logs; see `data/README.md`). PlatformIO's
default SPIFFS source dir is also named `data/`, which is exactly the
collision this rename avoids: `platformio.ini` points `[env:experiment]` at
`task_sequences/` via `board_build.filesystem_dir`.

`main_experiment.cpp` always loads `/experiment.json` from SPIFFS — there is
only one active payload per flash. To run a specific experiment, copy the
one you want over `experiment.json` before uploading the filesystem image:

```
cp task_sequences/coupling_cw.json task_sequences/experiment.json
~/.platformio/penv/bin/pio run -e experiment -t uploadfs
~/.platformio/penv/bin/pio run -e experiment -t upload
```

(`tools/run_coupling_sweep.py` does this copy + `uploadfs` step for you per
direction.)

## Files

- `experiment.json` — whatever's currently active (defaults to a copy of
  `test_experiment.json`, a safe no-op-ish sanity check).
- `test_experiment.json` — tiny 4-event sanity file for bench-testing the
  parser with the coil supply OFF (see plan verification step 1).
- `coupling_cw.json` / `coupling_ccw.json` — the coil-coupling
  characterization sweep (solo + pairwise + all-4, at several current
  levels), generated by `tools/gen_coupling_experiment.py`. Do not hand-edit;
  regenerate instead.
- `carrier_ramp.json` — replaces `main_carrier_ramp.cpp` (carrier duty 0→100%
  ramp at fixed 190Hz, phases `{0,180,90,270}`). That file used its own
  hardcoded GPIO pin map, different from `constants.h`'s current wiring —
  stale from an earlier board revision. This JSON reproduces the *drive
  behavior* through the current (correct) pins via `main_experiment.cpp`,
  not the old file's obsolete pin numbers.
- `takeoff.json` / `takeoff_upside_down.json` — replace `main_takeoff.cpp`
  (phases `{0,180,90,270}`, 100% carrier, 1→500Hz over 400s) and
  `main_takeoff_upside_down.cpp` (CW phases, 100% carrier, 1→190Hz over 40s).
  `takeoff.json` faithfully reproduces the original's back-to-back EASE-then-
  LINEAR ramp over the same 1→500Hz/400s range twice in a row, even though
  that looks like an unintentional duplication in the original file — this
  translation doesn't silently "fix" it.
- `dc_calibration.json` — approximates `main_DC.cpp` (which drove every PWM
  and carrier pin with a bare `digitalWrite(..., HIGH)`, bypassing
  `PWMController`/LEDC entirely) via 100% commutation duty + 100% carrier
  duty on all channels, which both park their respective pins statically
  HIGH through the normal driver path. Electrically equivalent for a DC
  calibration capture, but goes through LEDC-detach/timer logic instead of a
  raw register write, and (unlike the original, which stays on until
  manually reset) safely latches off after a 60s window — extend
  `duration_ms` if a longer calibration capture is needed.
- `comp_test.json` — replaces `main_comp_test.cpp`'s BASELINE (equal 50%
  duty) → GAP → TRIMMED (per-channel trim) A/B comparison, CCW, using the
  file's active `TRIM` values.
- `tilt.json` — a best-effort, simplified stand-in for `main_tilt.cpp`
  (itself already marked "OUTDATED, use tilt1+tilt2" in the old
  `platformio.ini`): a 1→210Hz ramp followed by a discrete 100%→0% duty
  sweep in 10% steps (2.5s holds), CCW. **Known gaps, not reproduced**: the
  live serial field-trim nudging (`a+`/`b-`/...) — that's interactive/
  closed-loop-adjacent and doesn't fit a pre-declared JSON queue — and the
  continuous frequency-blended trim schedule (`scheduledDutyPct`), which
  isn't expressible in the current JSON schema (no "value = f(live
  frequency)" construct). Also: the original file's checked-in state paired
  CCW phases with a CW-calibrated trim set (a mismatch, most likely leftover
  from editing); this JSON uses the correctly-paired CCW trim set
  (`comp_ccw_iter2`, {63.0, 100.0, 73.8, 63.7}) instead of reproducing that
  mismatch.
- **Not replaced**: `main_field_balance.cpp` (superseded — `main_current_pid.cpp`
  is its confirmed, more advanced successor for the same CS→ADC current-
  balance goal, not a "scripted experiment" to translate) and
  `main_test_current.cpp` (already failed to compile before this
  consolidation — stale/dead code, not a working experiment to preserve).
